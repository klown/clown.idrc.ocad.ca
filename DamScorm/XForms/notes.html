XForm Notes
Joseph Scheuhammer
April 19 2004

Consider the following HTML form, made up of radio button <input>'s, two text field <input>'s and a submit <input> (button):

Select Payment Method:

Cash Credit Card

Card Number:

Expiration Date:

XForm Parts

Divides the form into model and control:

Model:

<xforms:model>
 <xforms:instance>
 <ecommerce xmlns="ecom">
 <ecom:method/>
 <ecom:number/>
 <ecom:expiry/>
 </ecommerce>
 </xforms:instance>
 <xforms:submission action="http://example.com/submit" method="post" id="submit" includenamespaceprefixes=""/>
</xforms:model>

This says:

  • There are three parameters to the form, namely "method", "number", and "expiry".
    • Compare with HTML form example.
    • Note that the model says nothing regarding the UI for, these parameters.  The HTML form uses radio buttons, but the XForm model is neutral on this point.
  • The parameter names/elements are not part of the XForm namespace, but belong to a user-defined namespace (here the 'ecom' namespace).
  • The parameters are within an "xforms:instance".  More on that below, but, briefly, it allows for referencing these items in other parts of the document.
  • The last element, "xforms:submission", defines how the form contents are sent to the server.
  • The "xforms:instance" allows that the parameters could have intial or default values.
    • not done in the above example.
    • could be, for example, <ecom:method>cc</ecom:method> (meaning, use credit card for payment as the default value).
    • side effect:  no need for hidden elements as html forms sometimes require.  Simply put the "hidden" data into the xform model instance, and don't provide a control for it.

Control:

Method of Payment Control:

<xforms:select1 ref="method">
 <xforms:label>Select Payment Method:</xforms:label>
 <xforms:item>
 <xforms:label>Cash</xforms:label>
 <xforms:value>cash</xforms:value>
 </xforms:item>
 <xforms:item>
 <xforms:label>Credit</xforms:label>
 <xforms:value>cc</xforms:value>
 </xforms:item>
</xforms:select1>
  • "xforms:select1" is a control that says:
    • there are multiple possibilities.
    • only one possibility can be selected.
    • note, no committment to UI -- could be radio buttons, or menu, for example.
  • The possibilities are declared using "xform:item" elements.
  • xforms:label provide explicit label for the controls, however they are realized in the UI.
  • "xform:value" is the same as the value attribute typically found in html form elements.
    • e.g., <input type="radio" name="Cash" id="CashInput" value="cash"/>
Note the "ref" attribute to the select1 element.  This references back into the model above, and explicitly declares that this select1 control is for the method parameter of the model.

Controls for the other components of the model, namely, "number", "expiry", and "submission" are shown below.

Credit Card Number Control:

<xforms:input ref="number">
 <xforms:label>Credit Card Number:</xforms:label>
</xforms:input>

Expiry Date Control:

<xforms:input ref="expiry">
 <xforms:label>Expiration Date:</xforms:label>
</xforms:input>

Submit Control:

<xforms:submit submission="submit">
 <xforms:label>Submit</xforms:label>
</xforms:submit>

Note that the "ref" attribute here is replaced with a "submission" attribute.  Why?

Referencing

The example above connected, or bound, the controls to a model parameter via a "ref" attribute.  The examples used a simple ID technique for making the connection.  That is the ID "number" was used to bind the credit card number control with the model element named "number".

XForms allows other means of binding controls; in particular, it supports XPath as a way of finding the model component.  For example:

<xforms:input ref="/xforms:model/xforms:instance/ecommerce/number">...

If there were a way to identify the model component via an attribute, then one can use the XPath attribute syntax:  "@number".
