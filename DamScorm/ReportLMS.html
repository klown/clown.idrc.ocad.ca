Dynamic Appearance Model - Report on LMS
Joseph Scheuhammer
Feb 12, 2004

This document describes the rationale for choosing ADL's LMS for the DAM/SCORM project.

Focus of the Project

The main focus of this project is transformation of SCORM content in terms of its appearance.  Its goals are very similar to that of the TILE project.  The main difference is that TILE transformations are user-centric, whereas SCORM's can be institutionally based.  Nonetheless, a transformation is a transformation regardless of who it is for.

As such, I expect that much of what is currently done by TILE is either partially or completely applicable to the DAM. We (the ATRC) will spend much of our time extending our TILE efforts to satisfy the requirements of the DAM.  And, this is precisely where the bulk of the effort should go.

In this light, the LMS is of secondary importance.  However, there is a requirement that the learner interact with the content (lessons, SCO's, and other assets) using a SCORM-compliant LMS.  In an ideal world, the LMS would simply make requests of TILE for content, passing it the necessary transformation triggers.  TILE would fetch the content from a repository, transform it according to the transformation criteria, and send it back to the LMS.  The LMS would then present the result to the user.  With this architecture, in a sense, it does not matter what LMS is used, as long as it knows how to make requests of TILE.

In reality, nothing is actually that modular.  I suspect that we will have to both tinker with the LMS and TILE in order to make the two communicate properly.  Indeed, I expect a side result of this project is an API for TILE to interact with any LMS.  But, again, the focus should not be here; the effort should be the minimum needed to make TILE and the LMS talk profitably with one another.

LMS

We should use ADL's LMS for this project.  The reasons are:

  1. It is primarily written in Java, although some parts are ECMAScript.
    • The programmers working on this project are experienced in Java.  This decreases the learning curve.
    • The programmers also have some experience with ECMAScript.
    • The expectation then, is that any necessary modifications to the LMS should require minimal effort.
  2. It uses Apache's Jakarta Tomcat.
    • TILE uses Tomcat as well.
      • Again, this decreases the learning curve.
      • It likely decreases the amount of effort to construct a TILE/LMS interface.
  3. It uses Java Server Pages
    • In contrast, TILE uses Velocity templates.
    • There is a learning curve here, but it is minimal.
  4. Although it uses Java, Tomcat, ECMAScript and other cross-platform technology, it requires a Windows OS and the use of Internet Explorer.
    • This is primarily an annoyance in terms of debugging and testing.
    • It does require at least one Windows machine specifically for testing.  Since there are two developers on the project, it would be better if there were two test machines.
    • Development (writing code, compiling, quick experiments) can proceed on any machine since TILE itself is platform independent.
  5. ADL's LMS is guaranteed SCORM compliant.
