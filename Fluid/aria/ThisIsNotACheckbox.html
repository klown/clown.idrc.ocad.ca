This is not a checkbox.

The markup for the above "checkbox" is:

<h1 tabindex="0" class="checkbox"><img src="unchecked.gif"/> This is not a checkbox.</h1>

 This is an aria checkbox.

<h1 tabindex="0" class="checkbox" role="checkbox" aria-checked="false"><img src="unchecked.gif" role="presentation"/> This is an aria checkbox.</h1>

The problem: the markup alone does not describe the author's intent. Using CSS and JavaScript, an h1 element can look and behave like a checkbox.

One can argue that authors SHOULD NOT do this, warn them against doing so, and recommend that they use native roles instead. But, if they go ahead and do it anyway, there needs to be a way to communicate the proper semantics to assistive technologies. That's what ARIA is for:

  • ARIA is after-the-fact: the html/css/javascript has already repurposed the normal semantics.
  • ARIA does not affect how repurposed elements behave (the ultimate in unobtrusive?).

If it looks like a duck, and quacks like a duck, and waddles like a duck, well, then, what is it?

The h1/checkbox marked up with is spoken as a checkbox using JAWS10/FF3. The first one is not; it is spoken as a "heading level 1".
