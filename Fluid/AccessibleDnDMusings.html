Accessible Drag-and-drop Musings

Joseph Scheuhammer, April 12, 2007

One of TILE's authoring features, reorganizing learning objectives, could have been implemented using drag-and-drop. In fact, there was talk of writing JavaScript that would implement this dragging and dropping using a mouse and icons as is typically done. But, the accessibility conscience awoke and spoke, "hold on there -- is this GUI drag-and-drop thingy going to work with all users?" That led to an abandonment of the JavaScript implementation, and wondering about how to do it another way. That, in turn, led to a reflection on what drag-and-drop is. Here follows a characterization.

Starting with the GUI, drag-and-drop is a sequence:

  1. Using the mouse, select a source object (icon, block of text, etc.) by clicking on it.
  2. Hold down the mouse button while moving the mouse (the "drag").
    • Note: the visual feedback is a combination of the mouse cursor moving, and a "shadow" of the source object (e.g., dimmed icon) pinned under and following the mouse cursor.
  3. Move the mouse to a suitable target object (icon, text edit window, etc.)
    • Note: there is visual feedback that the target is suitable by changing its appearance (e.g., highlight), or not changing its appearance to indicate that it is unsuitable as a place to drop the source object.
  4. Drop the source object onto/into the target by releasing the mouse button.
    • If the target was suitable, there is visual feedback that shows the source being "absorbed" by the target -- e.g., the original source object and its shadow vanish, and the target is no longer highlighted.
    • If the target was unsuitable, there is visual feedback indicating that the drag-and-drop was cancelled -- e.g., the source object's shadow flys back to the original source, and the original source stays unchanged.

Next, abstract over the above sequence, ridding it of anything relating to the GUI; that is, forget the mouse, mouse button, mouse cursor movement, icons, shadows, and so on:

  1. Select a source object.
  2. Dragging is entirely GUI; ignore it.
  3. Select a target object.
  4. "Do it": confirm and effect the absorption of the source by the target.

Strictly speaking the above abstract characterization of drag-and-drop need not be a sequence. Users could select a target prior to selecting a source. Still, they cannot effect the move (step 4) until both source and target are selected.

To be slightly more explicit, here is what drag-and-drop boils down to:

  • There is a set of sources, one or more of which can be selected.
  • There is a set of targets, only one of which can be selected.
  • There is "do it" that moves the source to/into the target.

The following is the accessible implementation of drag and drop used in TILE. In this context, a course or lesson is made up of a tree of learning objectives. That is, there can be learning objectives that contain sub-objectives that, in turn, contain sub-objectives, and so on. The author can reorganize the objective hierarchy using an HTML form, the operation of which is explained within the example.

Organize Objectives

This page allows you to (re)organize your objectives.

The table below shows your current objectives. The left-most and right-most columns both show the current organization of your objectives.

  1. Select the objective you want to move from the left-hand column.
  2. Select how you want to move the objective using the middle column.
  3. Select the destination objective from the right-most column.
  4. Press the "Move Objective" button just below the table to move the objective.

The radio buttons in the middle column allow you to choose the insertion point relative to the destination. For example, if you choose "insert before...", then the objective to be moved will be placed immediately before the destination objective. Similarly with the other insertion points.

If the source objective has sub-objectives, they are moved as a group.

Some comments about the example. First, after submitting their declared reorganization, the page refreshes and shows the new organization in both the "Source Objective" and "Destination" columns. At this point users can proceed to perform further reorganization.

Secondly, the sources and targets are exactly the same heirarchy of learning objectives. This is mere happenstance that has to do with how TILE lessons are organized and nothing to do with the drag-and-drop in general. It is more likely that the source and target objects are different.

Thirdly, when an objective is moved, then all of its sub-objectives come along for the ride. This is analogous to dropping a folder onto/into another folder. It is not merely that the source folder is moved; all the files and sub-folders that are within it stay in their relative locations.

Thirdly, the fact that learning objectives are themselves ordered allows for moving them to different positions relative to the target. This is beyond most drag-and-drop scenarios. The most frequent use of drag-and-frop is where a user drags the source into the target container, without knowing where it will end up within that container. Hierarchies, or any other ordered collection, introduce some finesse where not only can the user put the source into the target, but at a specific position within it. The point is that frequently (when order is unimportant) the middle column of position options is unnecessary.

Finally, with respect to the suitability of the target in this example, it is illegal to move a source to a target when they are one and the same. In TILE, checking for suitability was handled by the server. That is, users could select impossible source-to-target reorganizations and submit them, but they would not take effect. The feedback that it was ineffectual was that the source and destination trees would appear exactly as before submission of the form. It would have been better if suitability had been handled on the client, say, by disallowing selection of a target objective if it was the same as the source objective.

To reiterate, the underlying functionality of drag-and-drop is:

  • select one or more sources from a collection of sources.
  • select one target.
  • execute the move of the source(s) to the target.

A final comment relates to the first point above regarding "one or more sources". There are contexts where only a single source object can be dragged. One can enforce this by using a user interface that allows only single selection. That can be done by a radio button group (see the TILE example above), or, if there are too many items to fit on a page, a list where the list allows only single selections.
